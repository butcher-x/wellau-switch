# 4.3 故障转移

## 功能说明

故障转移功能在主供应商请求失败时，自动切换到备用供应商，确保服务不中断。

**适用场景**：
- 供应商服务不稳定
- 需要高可用性
- 长时间运行的任务

## 前提条件

使用故障转移功能需要：

1. ✅ 启动代理服务
2. ✅ 开启应用接管
3. ✅ 配置故障转移队列
4. ✅ 开启自动故障转移

## 配置故障转移队列

### 打开配置页面

设置 → 高级 → 故障转移

### 选择应用

页面顶部有三个 Tab：
- Claude
- Codex
- Gemini

选择要配置的应用。

### 添加备用供应商

1. 在「故障转移队列」区域
2. 点击「添加供应商」
3. 从下拉列表选择供应商
4. 供应商会添加到队列末尾

### 调整优先级

拖拽供应商调整顺序：
- 序号越小，优先级越高
- 主供应商失败后，按顺序尝试备用供应商

### 移除供应商

点击供应商右侧的「移除」按钮。

## 主界面快捷操作

当代理和故障转移都开启时，供应商卡片会显示故障转移开关。

### 添加到队列

1. 找到供应商卡片
2. 开启故障转移开关
3. 供应商自动添加到队列

### 从队列移除

1. 关闭供应商卡片的故障转移开关
2. 供应商从队列中移除

## 开启自动故障转移

### 操作步骤

1. 在故障转移配置页面
2. 开启「自动故障转移」开关

### 开关说明

| 状态 | 行为 |
|------|------|
| 关闭 | 仅记录失败，不自动切换 |
| 开启 | 失败时自动切换到下一个供应商 |

## 故障转移流程

```mermaid
graph TD
    Start[请求到达代理] --> Send[发送到当前供应商]
    Send --> CheckSuccess{成功?}
    CheckSuccess -- 是 --> Return[返回响应]
    CheckSuccess -- 否 --> LogFail[记录失败]
    LogFail --> CheckCircuit{检查熔断状态}
    CheckCircuit -- 熔断 --> Skip[跳过此供应商]
    CheckCircuit -- 未熔断 --> IncFail[增加失败计数]
    Skip --> Next{队列中下一个?}
    IncFail --> Next
    Next -- 有 --> Switch[切换供应商]
    Switch --> Retry[重试请求]
    Retry --> Send
    Next -- 无 --> Error[返回错误]
```

## 熔断器配置

熔断器防止频繁重试失败的供应商。

### 配置项

不同应用有独立的默认配置。以下为通用默认值，Claude 有独立的宽松配置。

| 配置 | 说明 | 通用默认值 | Claude 默认值 | 范围 |
|------|------|--------|--------|------|
| 失败阈值 | 连续失败多少次触发熔断 | 4 | 8 | 1-20 |
| 恢复成功阈值 | 半开状态下成功多少次后关闭熔断器 | 2 | 3 | 1-10 |
| 恢复等待时间 | 熔断后多久尝试恢复（秒） | 60 | 90 | 0-300 |
| 错误率阈值 | 错误率超过此值时打开熔断器 | 60% | 70% | 0-100% |
| 最小请求数 | 计算错误率前的最小请求数 | 10 | 15 | 5-100 |

> 💡 Claude 由于请求耗时较长，默认配置更为宽松，容忍更多失败次数。

### 超时配置

| 配置 | 说明 | 通用默认值 | Claude 默认值 | 范围 |
|------|------|--------|--------|------|
| 流式首字节超时 | 等待首个数据块的最大时间（秒） | 60 | 90 | 1-120 |
| 流式静默超时 | 数据块之间的最大间隔（秒） | 120 | 180 | 60-600（填 0 禁用） |
| 非流式超时 | 非流式请求的总超时时间（秒） | 600 | 600 | 60-1200 |

### 重试配置

| 配置 | 说明 | 通用默认值 | Claude 默认值 | 范围 |
|------|------|--------|--------|------|
| 最大重试次数 | 请求失败时的重试次数 | 3 | 6 | 0-10 |

> 💡 Gemini 的默认最大重试次数为 5。

### 字段精确解释

以下解释基于当前代理实现。相关配置只有在「自动故障转移」开启时才会参与请求转发；关闭自动故障转移时，代理只使用当前供应商，并绕过故障转移的重试与超时逻辑。

#### 最大重试次数 vs 失败阈值

**最大重试次数**控制的是「同一条客户端请求最多可以尝试多少个供应商」。代码会把它转换为实际尝试上限：

```text
最大尝试供应商数 = 最大重试次数 + 1
```

例如：

```text
最大重试次数 = 0 → 最多尝试 1 家供应商
最大重试次数 = 3 → 最多尝试 4 家供应商
最大重试次数 = 6 → 最多尝试 7 家供应商
```

因此，Claude 默认 `6` 的含义是：一条请求失败后，最多可以沿故障转移队列尝试 7 家供应商。它不是熔断计数，也不会直接把某个供应商拉黑。

**失败阈值**控制的是「单个供应商连续失败多少次后进入熔断」。它按「应用类型 + 供应商」单独记录。例如 Claude 的供应商 A 连续失败达到 8 次时，A 会进入熔断状态；Codex 下同一个供应商的熔断状态不会和 Claude 共用。

简化理解：

```text
最大重试次数：这一条请求最多换几家供应商试
失败阈值：某一家供应商连续坏几次后先跳过它
```

例如故障转移队列为：

```text
A、B、C、D、E、F、G、H
```

当 `最大重试次数 = 6` 时，一条请求最多尝试：

```text
A → B → C → D → E → F → G
```

如果这 7 家都失败，会停止本次故障转移，不再尝试 H。与此同时，被实际尝试且发生可重试错误的供应商会各自记录一次失败；只有某个供应商自己的连续失败次数达到「失败阈值」时，它才会被熔断。

#### 哪些失败会计入熔断

只有被分类为「可重试」的错误才会记录到供应商健康度和熔断器，例如：

- 请求超时
- 连接失败、响应体读取失败等转发错误
- 流式静默超时
- 供应商认证、配置或格式转换相关错误
- 大部分上游 HTTP 错误，例如 `401`、`403`、`404`、`408`、`409`、`429`、`451` 和 `5xx`

以下错误被认为是客户端请求本身有问题，换供应商通常也无效，因此不会计入供应商熔断：

- `400 Bad Request`
- `405 Method Not Allowed`
- `406 Not Acceptable`
- `413 Payload Too Large`
- `414 URI Too Long`
- `415 Unsupported Media Type`
- `422 Unprocessable Entity`
- `501 Not Implemented`

成功请求会清空该供应商的连续失败计数。

#### 超时字段

**流式首字节超时**是等待上游返回第一个流式数据块的时间。上游返回响应头但迟迟不吐第一个数据块，也会按这个字段判定超时。

**流式静默超时**是流式响应已经开始后，两个数据块之间允许的最大间隔。它用于处理连接没有断开、但上游长时间不继续输出的情况。

**非流式超时**是非流式响应读取完整响应体的总等待时间。上游返回响应头后，如果响应体迟迟读不完，会按这个字段判定超时。

#### 熔断器设置

熔断器有三个状态：

```text
Closed    正常状态：允许请求
Open      熔断状态：跳过该供应商
HalfOpen  半开状态：只放少量请求试探恢复
```

**恢复等待时间**控制供应商进入 `Open` 后冷却多久才允许试探。例如设置为 90 秒时，供应商被熔断后的 90 秒内会被跳过；90 秒后会进入 `HalfOpen`。

**恢复成功阈值**控制 `HalfOpen` 状态下需要连续成功多少次才恢复为 `Closed`。例如设置为 3 时，供应商需要在半开试探阶段连续成功 3 次才会恢复正常。当前实现中，`HalfOpen` 同一时间只允许 1 个试探请求通过；如果试探失败，会立刻回到 `Open`，重新等待恢复时间。

**错误率阈值**用于处理「不是连续失败，但整体成功率很差」的供应商。计算方式为：

```text
错误率 = 失败请求数 / 总请求数
```

当错误率达到配置值时，供应商会进入熔断。例如错误率阈值为 70%，且已经满足最小请求数时：

```text
15 次请求中失败 11 次 → 11 / 15 = 73.3% → 触发熔断
15 次请求中失败 10 次 → 10 / 15 = 66.7% → 不因错误率触发熔断
```

**最小请求数**控制至少观察多少次请求后才开始计算错误率。例如设置为 15 时，即使前 3 次全部失败，也不会因为错误率 100% 立刻熔断；只有累计请求数达到 15 后，才会检查错误率阈值。

当供应商从 `HalfOpen` 恢复到 `Closed` 时，熔断器会重置总请求数和失败请求数，重新开始统计错误率。

### 熔断状态

| 状态 | 说明 |
|------|------|
| 关闭 | 正常状态，允许请求 |
| 开启 | 熔断状态，跳过此供应商 |
| 半开 | 尝试恢复，发送试探请求 |

### 状态转换

```mermaid
stateDiagram-v2
    [*] --> Closed: 初始化
    Closed --> Open: 失败次数 >= 阈值
    Open --> HalfOpen: 熔断时长到期
    HalfOpen --> Closed: 试探成功 (>= 恢复阈值)
    HalfOpen --> Open: 试探失败
```

## 最佳实践

### 队列配置建议

1. **主供应商**：最稳定、最快的供应商
2. **第一备用**：次优选择
3. **第二备用**：保底选择

### 熔断器配置建议

| 场景 | 失败阈值 | 熔断时长 |
|------|----------|----------|
| 高可用要求 | 2 | 30 秒 |
| 一般场景 | 3 | 60 秒 |
| 容忍偶发失败 | 5 | 120 秒 |

### 监控建议

定期检查：
- 各供应商的健康状态
- 故障转移发生频率
- 熔断触发情况

## 常见问题

### 故障转移没有触发

检查：
1. 代理服务是否运行
2. 应用接管是否开启
3. 自动故障转移是否开启
4. 队列中是否有备用供应商

### 频繁触发故障转移

可能原因：
- 主供应商不稳定
- 网络问题
- 配置错误

解决方法：
- 检查主供应商状态
- 调整熔断器参数
- 考虑更换主供应商

### 所有供应商都熔断

等待熔断时长到期后自动恢复，或：
1. 手动重启代理服务
2. 重置熔断状态
